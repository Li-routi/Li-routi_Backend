# 테스트 컨벤션

## 목적과 범위

이 저장소의 단위 테스트, 슬라이스 테스트, 통합 테스트가 일관된 경계와 스타일을 갖도록 한다.  
이 규칙은 새로 작성하거나 수정하는 테스트에 적용한다. 기존 레거시 테스트는 포맷 정리만을 이유로 다시 쓰지 않는다.

## 기본 구조

- 테스트는 Arrange-Act-Assert 순서로 작성하고, 구간을 `// given`, `// when`, `// then` 주석으로 표시한다.
- 주석은 짧게 유지하고, 시나리오는 메서드명과 `@DisplayName`으로 설명한다.
- 한 테스트는 하나의 행동 또는 하나의 실패 경로에만 집중한다.
- 가능하면 테스트 대상은 한 번만 호출한다. 반복 호출은 그것 자체가 계약일 때만 허용한다.

## 명명 규칙

- 단위 테스트와 서비스 테스트는 `<Subject>Test` 형식을 사용한다.
- 테스트 경계가 중요하면 `<Subject>WebTest`, `<Subject>RepositoryTest`, `<Subject>IntegrationTest`처럼 구분자를 붙인다.
- 메서드명은 가능하면 `<method>_<scenario>_<expectedResult>` 형식으로 작성한다.
- `@DisplayName`은 관찰 가능한 결과를 짧고 읽기 쉽게 설명한다.

## 단언 규칙

- 결과 검증은 `then`에 둔다.
- 여러 검증이 하나의 결과를 설명할 때는 `assertAll`을 사용한다.
- 예외 테스트에서는 `when`에 `assertThrows`로 예외를 잡고, `then`에서 도메인 코드나 의미 있는 필드를 검증한다.
- `assertDoesNotThrow`는 실패가 없다는 점 자체가 주요 계약일 때만 사용한다.
- 단언은 내부 구현이 아니라 관찰 가능한 결과, 상태, 에러 코드를 검증한다.
- 컬렉션 순서가 계약이 아닌데도 순서에 의존하는 단언은 피한다.

## Mockito 규칙

- 일반 Mockito 테스트에서는 `@ExtendWith(MockitoExtension.class)`를 사용한다.
- 저장소, 외부 클라이언트, 인프라 유틸리티는 격리가 필요할 때만 mock으로 둔다.
- 순수 파서, 검증기, 변환기, 결정적 생성기는 가능한 실제 객체를 사용한다.
- stub은 시나리오에서 실제로 사용하는 동작만 설정한다. 특별한 이유 없이 `lenient()`를 쓰지 않는다.
- 계약에 중요한 인자는 정확한 값 또는 `eq()`를 우선 사용한다. 의미 없는 값에만 `any()`를 쓴다.
- 협력 객체에 전달된 내용이 검증 대상이면 `ArgumentCaptor`를 사용한다.
- `verify`와 `never()`는 계약상 중요한 상호작용에만 사용한다.
- `verifyNoMoreInteractions`는 관성적으로 추가하지 않는다.

## 테스트 유형

### 순수 도메인 및 서비스 단위 테스트

- 생성한 실제 객체나 Mockito만으로 충분하면 Spring 컨텍스트를 올리지 않는다.
- 반환값, 상태 변화, 도메인 예외, 중요한 협력 객체 상호작용을 검증한다.
- 성공, 검증 실패, 소유권, 없음(not-found), 의존성 실패 경로를 상황에 맞게 커버한다.

### 컨트롤러 슬라이스 테스트

- 요청/응답 계약 테스트에는 `@WebMvcTest`, `MockMvc`, `@MockitoBean`을 사용한다.
- 상태 코드, 응답 코드, 대표 JSON 필드, 검증 실패, 서비스 호출을 검증한다.
- 요청 데이터와 서비스 stub은 `given`에, `perform`은 `when`에, 기대값은 `then`에 둔다.
- 보안 필터 비활성화는 컨트롤러 계약 테스트에만 허용된다. 보안 자체는 검증하지 않는다.
- 익명 접근, 인증, 인가, 잘못된 토큰은 별도의 필터 활성화 테스트로 검증한다.

### 저장소 테스트

- 커스텀 쿼리, 정렬, 소유권 조건, 매핑, 제약 검증에는 `@DataJpaTest`를 사용한다.
- 실제 동작 차이가 날 수 있으면 임베디드 DB보다 실제 MySQL 방언을 우선한다.
- DB 읽기나 제약 검증을 볼 때는 persistence context를 flush/clear 한다.
- 필요하면 N+1이나 lazy-loading 가정을 함께 확인한다.

### 통합 테스트

- 여러 애플리케이션 계층이나 실제 인프라가 함께 동작해야 할 때만 `@SpringBootTest`를 사용한다.
- MySQL 또는 Redis 의존 테스트는 문서화된 로컬 Docker 서비스나 명시된 컨테이너 기반 환경을 사용한다. H2를 가정하지 않는다.
- 테스트 전용 설정과 환경 변수를 사용한다. 실제 자격 증명이나 `.env` 값을 직접 넣지 않는다.
- 테스트 데이터는 rollback, 명시적 정리, 또는 범위가 분리된 고유 데이터로 격리한다.
- 필요한 테스트를 장기간 `@Disabled`로 두지 않는다. 이유와 복구 조건을 함께 기록한다.

### 예외 및 Advice 테스트

- 직접 advice 테스트는 예외와 응답 코드 매핑을 최소한의 대표 데이터로 검증한다.
- HTTP 상태, 직렬화, validation, resolver 동작이 계약이면 MockMvc를 사용한다.

## 픽스처와 결정성

- 픽스처는 최소한으로, 명시적으로 유지한다. 반복은 helper 메서드로 처리하되 precondition을 숨기지 않는다.
- `ReflectionTestUtils`는 생성된 ID나 audit timestamp처럼 persistence-managed 필드에만 제한한다.
- 시간이 검증에 영향을 주면 `now()`보다 고정된 값이나 `Clock`을 사용한다.
- 동기화를 위해 `Thread.sleep()`을 사용하지 않는다. 결정적 방식이나 bounded waiting을 사용한다.
- 반복되는 파서, 검증기, 경계값 테스트에는 `@ParameterizedTest`를 사용한다.
- 순차적인 mock 응답만으로 동시성을 증명할 수 없다. 동시성이 계약이라면 DB나 Redis atomicity를 적절한 통합 테스트로 검증한다.

## 검증

- 작업 중에는 가장 가까운 관련 테스트를 먼저 실행한다.
- 손이 닿는 범위가 크면 손보기 전에 `./gradlew test`, 범위가 넓으면 `./gradlew clean build`를 실행한다.
- 통합 테스트가 MySQL이나 Redis를 필요로 하면, 실행 전에 문서화된 서비스를 먼저 띄운다.
- 실행하지 못한 테스트는 생략 사유와 남는 리스크를 함께 보고한다.
- 실패한 테스트를 통과시키려고 우회하거나 약화하지 않는다.
